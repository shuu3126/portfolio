PROJECT 2

サービスLP

短時間で価値を伝え、CVへ誘導

GOAL

ゴール / KPI

  • 一次KPI： CTAクリック率 / 到達率（Hero → 料金 → フォーム）
  • 二次KPI： 問い合わせ送信数 / 資料DL数
  • 補助指標： 直帰率・平均滞在・スクロール深度
STRUCTURE

セクション構成 / ユーザー心理

  1. Hero：価値提案＋CTA（関心・期待）
  2. Benefit：3つの便益（納得）
  3. Detail：詳細/機能（理解）
  4. Social Proof：実績・声（安心）
  5. 料金・FAQ：迷いの解消（意思決定）
  6. 最終CTA：アクション（申込/問い合わせ）
COPY

コピー戦略

  • 見出しは1画面1メッセージ、副題で反論への一次回答を添える
  • CTA文言は「行為＋価値」（例：無料で相談する / 1分で問い合わせ）
  • FAQは問い合わせ履歴を前提に「なぜ？」に答える形式で作成
DESIGN

デザイン指針

  • 余白と階層でスキャン性を担保、カード化で情報ブロックを明確化
  • 配色は白/黒/アクセント1色、CTAは常にコントラスト比4.5以上
  • 視線の流れ：左上→右下を意識し、CTA直前にベネフィットを配置
SPEC

実装 / 計測

  • HTML/CSS/JS（バニラ）。画像は遅延読込、ヒーローは優先読込
  • イベント計測（CTAクリック・フォーム入力開始・送信完了）
  • ABテスト前提でコピー/CTAを差し替えやすいDOM構造に分離
QUALITY

品質（SEO / A11y / Perf）

  • SEO：メタ/見出し/内部リンク、ファーストビューに主要KWを自然配置
  • アクセシビリティ：代替テキスト・ラベル・フォーカスリング・タブ順
  • パフォーマンス：CLS 0.1以下、LCP 2.8s以下、不要JS排除
MY WORK

担当範囲 / 実施内容

  • ページ設計〜制作： 情報設計・ワイヤー・UIコンポーネント設計からHTML/CSS/JS実装まで一気通貫で担当。
  • 使用画像の作成・デザイン： ヒーロー/セクション用のビジュアルを用途別に最適化（解像度・比率・圧縮）。
  • DBの管理・適用・実装： 基本的なテーブル設計・接続・CRUD処理を実装し、LPからのデータ連携を整備。
  • 問い合わせ導線： 公式LINEの開設（問い合わせで使用）と、ヘッダー画像・アイコンのデザイン制作。
CHALLENGES

行き詰まった点 / 困ったことと対処

  • 初めてのDB実装でトラブル多発
    難所： 接続まわり/エンコード/権限設定/スキーマ更新の整合性。
    対処： 役割ごとにテーブルを最小構成で分割 → マイグレーション手順をメモ化 → 接続/例外処理を共通関数に集約。
    結果： 接続〜CRUDの失敗ポイントを切り分けしやすくなり、改修の影響範囲を限定。
  • ログイン機能の実装が難しかった
    難所： セッション管理・バリデーション・リダイレクト制御・エラーハンドリング。
    対処： 「未ログイン/ログイン済み」でミドルウェア的チェックを最初に挟み、処理を早期分岐。CSRF対策・入力サニタイズを基本セットに。
    結果： 認証まわりの分岐が明確になり、想定外の画面遷移を抑制。
  • ゲスト購入とログイン購入の分岐設計（PHP + DB 初心者には大きな壁）
    難所： 同一の「購入」でも状態により必要画面・処理が異なる（ゲスト/ログイン/途中ログイン）。
    対処： 購入フローを状態遷移図に落とし込み、guest → login → checkout の3系統をルーティングで明確化。セッションに「進捗フラグ」を持たせて戻り先/次遷移を一元管理。
    結果： 迷子になりやすい分岐を解消し、テストケースの網羅がしやすくなった。
  • デザイン要件が自由度高すぎる
    難所： 依頼主からの詳細指示がなく、方向性の意思決定コストが高い。
    対処： LPの目的に立ち返り、白/黒/アクセント1色＋カード化＋階層強調で読みやすさとCV導線を優先。CTAは常に高コントラストに。
    結果： 自由度の高い中でも一貫したトーン&マナーを維持し、内容が乗る器を先に固められた。
DECISIONS

主な意思決定（設計 / コピー / UI）

  • 採用： 「1画面1メッセージ」＋直後に一次反論への回答（迷いの削減／CV直結）
  • 採用： CTA文言は「行為＋価値」（例：無料で相談する、1分で問い合わせ）で具体化
  • 採用： 見出し/本文/注記の3段階のタイポ階層とカードレイアウトでスキャン性を担保
  • 見送り： 色の多用・装飾過多（可読性とパフォーマンスを優先）
  • 見送り： 早期の複雑な推奨商品ロジック（DB学習段階では保守性を優先）
RETROSPECT

振り返り / 次の改善

  • FAQの順序を検索クエリに合わせて最適化（実データ連動）
  • ヒーローの背景画像を季節替えにして再訪時の新規性を担保
